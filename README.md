# viaPhoton — Nut Transport (Jeep Problem)

## 1. What it does

A Jeep / Desert-Crossing solver: given `D`, `N`, `F`, `C` (distance, pile size, fuel per km, cart capacity), compute `X`, the maximum kilograms of nuts deliverable to the town. Two entry points share one pure core — a Next.js web UI and a Node CLI.

## 2. Run it

```bash
pnpm install

# web UI
pnpm dev                                # http://localhost:3000

# CLI
echo "10,100,1,10" | pnpm cli           # one-shot
pnpm cli < input.txt                    # batch of lines
```

Other useful scripts:

```bash
pnpm test            # vitest
pnpm test:coverage   # 100% gate
pnpm typecheck       # tsc --noEmit
pnpm lint            # biome
pnpm build && pnpm start
```

## 3. Algorithm

The brief states the cart carries at most `C` kg and the horse burns `F` kg/km **from the cart's contents** regardless of load. The hint — *"`X > 0` even if `D·F > C`"* — exists to defeat the naive answer `X = max(0, N − D·F)`. When a single load can't reach the town, the optimal play is a **staged-depot shuttle**: drive forward, deposit, drive back empty (the cart's fuel reserves cover the return), reload, repeat.

At position `p` with mass `m`, let `k = ⌈m / C⌉` (trips needed). To advance `k` loads forward by distance `d`, the cart makes `k` forward trips and `k − 1` return trips — so `2k − 1` traversals consuming `(2k − 1)·F·d` total fuel. Setting `d_k = (m − (k − 1)·C) / ((2k − 1)·F)` consumes exactly one full load and leaves `k − 1` full loads at the new depot. Recurse until `mass ≤ C`, then drive the final leg one-way.

The canonical anchor: `solve(10, 100, 1, 10) ≈ 13.997665904786647`. A correct implementation produces ~14 here, not 0 — this single number distinguishes the staged-shuttle solution from every naive variant. The test suite pins it as a closed-form cross-check.

## 4. Decisions and tradeoffs

- **Next.js + TypeScript + Zod** — modern stack literacy; type safety from input to output; Zod is the only validator so there are no parallel hand-typed DTOs.
- **shadcn/ui (new-york / neutral)** — zero hand-rolled UI primitives, polish without bespoke CSS. Vendored components are excluded from our coverage gate; they're not ours.
- **Hexagonal-lite layout** — `lib/` is pure domain + application; `components/` and `scripts/` are adapters. Function signatures are the ports; no `interface SolveUseCase` ceremony for a 35-line algorithm.
- **No `react-hook-form`** — one textarea, one submit. A form library here would be added surface area to cover for zero gain.
- **CLI alongside the web UI** — the spec says command-line *or* textarea; offering both is essentially free since `scripts/cli.ts` is ~10 lines of stdin/stdout glue around the same `lib/process.ts` the browser calls.
- **100% coverage gated in `vitest.config.mts`** — not aspirational, enforced. Drops in coverage fail the run.

## 5. What's missing / what I'd do with more time

- Playwright E2E covering the dev server, not just the React tree.
- Server-side parsing with streaming feedback while the user types.
- A11y audit beyond the obvious labels/roles.
- A property-based test (fast-check) generating random `(D, N, F, C)` and checking invariants (monotone in `N`, scaling identity).
- Containerized run target — multi-stage Dockerfile + compose for a one-command demo.
- i18n if the audience extends beyond English.
